# Interface Taxonomy Playbook

Use this playbook to classify the surface before choosing layout, density, motion, and verification criteria.

## Classification Steps

1. Identify the user's primary intent: learn, buy, compare, operate, create, play, book, review, or monitor.
2. Identify session length: glance, short task, repeated shift, long editing session, exploration, or game loop.
3. Identify information density: sparse persuasion, medium content, dense data, canvas/tool, or real-time state.
4. Pick one primary mode. Secondary modes can appear, but they must not control the first viewport.

## Modes

### Marketing / Landing

Goal: convert interest into one next action.

Structure:

- brand/product signal;
- clear offer;
- proof or artifact;
- primary CTA;
- supporting section hint;
- objections/evidence/FAQ lower on page.

Common misuse:

- abstract hero with no product artifact;
- three generic cards as the whole argument;
- stock-like imagery hiding the real product.

Acceptance:

- first viewport explains category and next action;
- proof is concrete;
- next section is visible or clearly reachable;
- mobile CTA remains accessible.

### Product Website

Goal: explain a product and help users evaluate fit.

Structure:

- product artifact or screenshot;
- capabilities tied to user tasks;
- comparison/proof;
- pricing/demo/contact path if applicable;
- docs or detail paths.

Common misuse:

- landing-page hype without product specificity;
- feature copy detached from UI evidence.

Acceptance:

- users can inspect what the product is;
- copy names real objects and outcomes;
- visual system reinforces product category.

### Web App / SaaS

Goal: help users perform work.

Structure:

- app shell;
- navigation;
- current task surface;
- lists/tables/cards as appropriate;
- action feedback;
- state handling.

Common misuse:

- opening with a marketing hero;
- decorative cards instead of controls;
- no empty/error/disabled states.

Acceptance:

- first screen is usable;
- controls have feedback;
- keyboard and mobile paths are considered.

### Dashboard / Operational Console

Goal: scan, compare, monitor, and act repeatedly.

Structure:

- status summary;
- exception/high-risk lane;
- table/list/map/timeline;
- filters and time controls;
- pinned actions;
- drill-down path.

Common misuse:

- oversized hero typography;
- marketing sections;
- chart decoration without operational meaning.

Acceptance:

- abnormal states are easiest to spot;
- dense data remains readable;
- repeated-shift use feels calm and fast.

### Editor / Workspace

Goal: create or manipulate an artifact.

Structure:

- canvas or work surface;
- toolbar;
- side panels;
- selection state;
- history/undo/save;
- inspector/detail controls.

Common misuse:

- hiding tools below decorative content;
- unstable layout when panels open;
- no selected/dirty/saving state.

Acceptance:

- work surface is primary;
- controls are predictable;
- layout stays stable during editing.

### Portfolio / Personal Site

Goal: show taste, proof, and navigation through work.

Structure:

- identity signal;
- project index;
- selected case studies;
- process/proof;
- contact path.

Common misuse:

- visual effects that obscure work;
- no project evidence;
- novelty navigation that blocks scanning.

Acceptance:

- work is inspectable;
- personality is specific;
- motion does not slow reading.

### Ecommerce / Content Site

Goal: browse, inspect, filter, buy, subscribe, or read.

Structure:

- product/article grid or feed;
- filters/search;
- detail view;
- trust and metadata;
- cart/subscription/related paths.

Common misuse:

- generic lifestyle hero instead of product inspection;
- missing variants, states, or metadata.

Acceptance:

- users can compare items;
- important metadata is visible;
- mobile browse and detail paths work.

### Mobile H5 / WeChat Mini Program

Goal: finish a short task in a mobile context.

Structure:

- primary task entry;
- shallow navigation;
- thumb-friendly controls;
- concise labels;
- persistent action when appropriate.

Common misuse:

- desktop landing page layout;
- long scroll narratives;
- tiny tap targets.

Acceptance:

- task is visible immediately;
- touch targets are at least 44px;
- unavailable/loading/error states are clear.

### Game / Interactive Toy UI

Goal: support play and immediate feedback.

Structure:

- game surface;
- score/progress/state;
- controls;
- feedback;
- pause/restart/settings.

Common misuse:

- UI chrome competing with play;
- unclear controls;
- animations hiding state changes.

Acceptance:

- player state is readable;
- controls fit input mode;
- feedback is immediate and does not block play.
